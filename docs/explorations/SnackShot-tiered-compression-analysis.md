# 나이 기반 단계 압축(Tiered/Aged Compression) — 타당성·설계·비용 분석

> 아이디어: 클립을 나이(월 단위 등)에 따라 단계적으로 더 강하게 압축. 최근=고화질, 오래될수록 용량↓.

## 0. 현재 상태 / 문제
- 녹화 720p → 압축 `maxSize 1280(=720p) @ 3Mbps` → **해상도·비트레이트 거의 그대로** 재인코딩 → 압축률 낮음(41MB→34MB).
- 총량의 대부분은 **원본(ADR-004로 보존)**.

## 1. 평가 — 합리적인가?
**예.** 다이어리는 최근 클립을 자주 보고 오래된 건 보관 목적이므로, 시간축으로 화질-용량을 최적화하는 건 타당. 단 아래 3가지가 핵심 제약.

## 2. 핵심 주의점 (설계를 좌우)
1. **세대 손실(generation loss)**: 이미 압축된 파일을 또 압축하면 화질이 누적 저하된다.
   → 각 단계는 **반드시 원본에서 재인코딩**해야 손실 최소화. 즉 *단계 압축의 전제 = 원본 보존*.
   → 그런데 마지막 단계에서 "원본 삭제"를 하면 더는 재인코딩 불가. **원본 삭제는 최종 단계에서만**, 그 시점 이후엔 추가 단계 없음.
2. **vault 재동기화**: 옵시디언으로 export된 미디어는 압축본 복사본이다. 재압축하면 vault 미디어가 stale → **해당 날 재export 필요**(추가 복잡도·쓰기).
3. **CPU/배터리/시간**: 로컬 재인코딩은 API 비용 0이지만 기기 자원을 쓴다 → **충전 중 + 유휴 시 배치 실행** 권장.

## 3. 단계 설계 (예시 — 값 조정 가능)
| 단계 | 나이 | 해상도 | 비트레이트 | 원본 |
|------|------|--------|-----------|------|
| S1 | 0~1개월 | 720p (maxSize 1280) | 3 Mbps (현행) | 보존 |
| S2 | 1~3개월 | 540p (maxSize 960) | 1.5 Mbps | 보존 |
| S3 | 3~6개월+ | 480p (maxSize 854) | 1 Mbps | (옵션) 삭제 |

- 각 단계 전이 시 **원본 → 타깃 스펙 재인코딩 → 압축본 교체**.
- "녹화 화질 낮추기"는 기존 클립엔 적용 불가(캡처 시점 결정) → 기존 클립의 S3는 "가장 공격적 재압축 + 원본 삭제"로 해석.

## 4. 구현에 필요한 변경
- **DB(마이그레이션 v8)**: `entries.compression_tier INTEGER DEFAULT 1` (현재 단계 추적).
- **잡 타입 추가**: `recompress` (`AiJobType` union + ai_jobs CHECK 제약 갱신).
- **스케줄러**: `expo-background-fetch`(이미 의존성 보유) 또는 앱 시작/포커스 시 "다음 단계 대상(나이·tier 기준)" 스캔 → `recompress` 잡 큐잉. 기존 단일 워커로 순차 처리.
- **핸들러 `handleRecompress`**: 원본에서 단계 스펙으로 재인코딩 → 압축본·썸네일 교체 → `compression_tier` 갱신 → (S3) 원본 삭제 → vault 재export 큐잉.
- **설정 UI**: 단계 정책 on/off, 원본 삭제 허용 토글, 충전 중에만 실행 토글.
- **ADR**: ADR-004(원본 보존)·ADR-022(압축 스펙) 개정 또는 신규 ADR("나이 기반 단계 압축").

## 5. 비용
- **API 비용 0**(로컬 인코딩). 
- 자원 비용: 재인코딩 CPU/배터리. 클립당 수 초~수십 초. 배치로 분산.
- 저장 절감: S2까지만 적용해도 압축본 ~50%↓. S3+원본삭제까지면 총량 대폭↓(원본이 가장 큰 덩어리이므로).

## 6. 복잡도 평가
- 단계 압축은 효과적이지만 **상태(tier)·스케줄러·재인코딩·vault 재동기화·ADR 변경**까지 걸치는 **본격 기능**(중상).
- 세대 손실/원본 삭제 타이밍을 잘못 잡으면 화질 회복 불가 → 신중.

## 7. 권장 — 단계적 도입(ROI 순)
1. **즉효·저복잡 (먼저)**: 기본 압축 스펙만 강화 — 신규 클립부터 `maxSize 960 @ 1.5Mbps`. 코드 한 곳(handlers) 변경, 위험 0. 압축본 즉시 ~절반.
2. **원본 자동 정리(옵션 토글)**: "N개월 지난 클립의 원본 삭제"를 설정 토글로. 총량 최대 절감, 구현 단순(스캔+삭제, 재인코딩 불필요). ADR-004는 "옵션으로 사용자 선택" 형태로 개정.
3. **본격 단계 압축(원하면)**: 위 4번 전체. tier 컬럼 + recompress 잡 + 배치 + vault 재export. 화질-용량을 시간축으로 정교하게 최적화하고 싶을 때.

## 8. 제안
- **1단계(스펙 강화)**를 지금 적용 → 신규 클립부터 바로 체감.
- **2단계(원본 N개월 후 정리 옵션)**를 그다음 — 기존 클립 총량까지 절감.
- **3단계(월단위 다단계 재압축)**는 위 둘로도 부족하거나, "최근은 선명·오래된 건 작게"를 꼭 원할 때 본격 진행.

이렇게 하면 복잡도/위험을 낮추면서 절감 효과의 대부분을 먼저 확보하고, 진짜 단계 압축은 필요할 때 얹을 수 있습니다.
