# VAD로 무음 클립 STT 전송 건너뛰기 — 타당성/구현/영향/비용 분석 (구현 전 조사)

> 목적: 무음·저음성 클립에서 Whisper 환각("뜬금없는 대사")을 막기 위해, **STT API 호출 전에 음성 유무를 판정(VAD)해 전송을 건너뛰는** 방안의 적절성·난이도·설계 영향·비용을 정리한다. (구현 X)

## 0. 현재 상태 (참고)
- 이미 **사후 환각 필터** 적용됨(`whisper.ts`): `temperature=0` + 세그먼트 단위 `no_speech_prob`/`avg_logprob`/`compression_ratio` 기준으로 환각 구간 제거 → 가짜 문장은 본문에 남지 않음.
- 즉 **정확도(가짜 텍스트 저장 방지)는 이미 상당 부분 해결**. VAD의 추가 가치는 "전송 자체를 건너뛰어 비용·낭비를 줄이는 것"에 가깝다.

## 1. 결정적 제약 — 모드별 오디오 접근성
| 모드 | 녹음 수단 | 실시간 레벨(metering) | 사후 분석 |
|------|-----------|----------------------|-----------|
| 음성(audio) | expo-audio | **가능** (`isMeteringEnabled`, status.metering dB) | 파일 디코딩 필요 |
| 영상/독백(voice) | expo-camera `recordAsync` | **불가** (오디오 레벨 API 없음) | mp4 오디오 트랙 디코딩 필요 |

→ **환각이 실제로 문제되는 건 voice(영상) 모드인데, 거기엔 실시간 metering이 없다.** 따라서 "녹음 중 레벨로 판정"은 audio 모드에만 적용 가능하고, voice 모드까지 덮으려면 **녹화 후 파일을 디코딩**해야 한다.

## 2. VAD 방식 분류 (정확도 ↑ = 비용 ↑)
1. **에너지/무음 감지(silence detection)**: RMS/dB 임계. 말 vs 잡음 구분 못 함(시끄러운 무음은 통과). 가볍다.
2. **WebRTC VAD**: 프레임 단위 음성/비음성. 가볍고 실시간. C 라이브러리 → RN 바인딩 필요.
3. **신경망 VAD(Silero 등)**: 정확. onnxruntime 필요(번들 큼).

대부분의 경우 1~2로 충분. 환각은 "거의 무음"에서 나므로 에너지 기반으로도 큰 효과.

## 3. 구현 옵션 비교
### A. 녹음 중 metering 게이트 (audio 모드 한정)
- expo-audio `isMeteringEnabled` 켜고 녹음 내내 peak/avg dB 추적 → 임계 미만이면 "무음"으로 보고 STT 잡을 enqueue하지 않음(또는 `stt_status='skipped'`).
- **장점**: 새 네이티브 의존 0, 매우 쉬움, 무료, 즉시 적용.
- **단점**: **voice(영상) 모드 미적용**(가장 문제되는 모드를 못 막음). 에너지 기반이라 잡음=음성 오판 가능.
- 작업량: 소(record-audio.tsx + preview-audio.tsx 흐름에 플래그 전달).

### B. 사후 파일 디코딩 + 에너지/VAD (voice 포함 전체 커버)
- 저장된 원본(mp4/m4a)의 오디오 트랙을 디코딩(PCM) → RMS 에너지 또는 WebRTC VAD로 음성 유무 판정 → handleStt에서 무음이면 `skipped`.
- 디코딩 수단:
  - **커스텀 네이티브 모듈**: Android `MediaExtractor`+`MediaCodec`, iOS `AVAssetReader`로 PCM 추출 후 RMS 계산(Kotlin/Swift 직접). 의존성 추가 없음, 번들 영향 적음, 그러나 **양 플랫폼 네이티브 코드 작성 필요**.
  - **ffmpeg-kit**: 디코딩 만능이나 **프로젝트 은퇴(유지보수 종료)** + 바이너리 대용량 → 비권장.
  - **onnxruntime-react-native + Silero**: 정확하지만 모델/런타임으로 번들 수십 MB↑ → 개인용엔 과함.
- **장점**: voice 모드까지 일관 적용.
- **단점**: 네이티브 모듈 + config plugin 필요, 임계값 튜닝, 디코딩 시간(짧은 클립은 무시할 수준), 유지보수 부담.
- 작업량: 중상(커스텀 네이티브 모듈 기준).

### C. 현행 사후 환각 필터 유지 (추가 작업 0)
- 이미 적용됨. 비용 절감은 없지만 "가짜 텍스트 저장"은 막음.

## 4. 설계 영향 (B/A 공통)
- **트리거 위치**: STT enqueue 지점(preview/preview-audio) 또는 `handleStt` 진입부에 "무음이면 skip" 분기.
- **상태값**: 무음 skip 시 `stt_status='skipped'` 재사용(스키마 변경 불필요). 단 "실패"가 아니라 "건너뜀"이므로 UI 문구 구분 필요(예: "음성 없음").
- **연쇄 작업**: STT skip → `label_extraction`도 skip(현재 흐름은 stt done/skipped일 때 라벨 큐잉; skip이면 텍스트 없어 자동으로 후보 0 → 사실상 무해). 옵시디언 export는 빈 본문으로 진행.
- **ADR-002(STT 인터페이스 추상화)**: VAD는 STT 앞단 별도 단계로 두는 게 깔끔(예: `VadService` 또는 `handleStt` 내부 전처리). 인터페이스 1개 추가.
- **Today/상세 표시**: 무음 클립은 "트랜스크립트 없음" 대신 "음성 없음(건너뜀)"으로 안내하면 사용자가 환각 제거와 구분 가능.
- **오판 위험**: 임계 너무 높으면 진짜 발화도 skip → 데이터 손실. 보수적 임계 + "재시도(STT)" 버튼(이미 상세 화면에 있음)으로 복구 경로 확보.

## 5. 비용 분석
- Whisper 단가 **$0.006/분**. 무음 클립을 건너뛰면 그 클립의 호출만 절약.
- 개인용 다이어리(하루 수 건, 대부분 실제 발화)에선 **무음 클립 비중·총 호출량이 작아 절감액이 미미**(월 몇 센트 수준).
- 즉 **비용은 VAD 도입의 주된 정당화 근거가 되기 어렵다.** 진짜 동기는 "환각 방지"인데 그건 사후 필터로 이미 큰 부분 처리됨.
- 반대로 B(네이티브 모듈)는 **개발·유지보수 비용**이 절감액을 크게 초과.

## 6. 권장안 (ROI 순)
1. **현행 사후 필터 유지 + 마감 처리(권장, 추가비용 최소)**
   - 환각 필터 통과 후 **본문이 비면 `stt_status='skipped'`로 표기**하고 UI를 "음성 없음"으로 → 빈 트랜스크립트가 깔끔해지고 사용자 혼란 제거. (네이티브 불필요, 작은 코드 변경)
   - 필요 시 환각 임계값만 실제 클립으로 튜닝.
2. **audio 모드 한정 녹음-중 metering 게이트(선택, 쉬움)**
   - expo-audio metering으로 완전 무음 녹음은 애초에 저장/전송 안 함. voice 모드는 미적용.
3. **voice까지 커버하는 사후 네이티브 VAD(보류 권장)**
   - 커스텀 MediaCodec/AVAssetReader 모듈로 RMS 무음 판정. 정확도·일관성은 최고지만 **작업량·유지보수 대비 효용이 개인용엔 낮음.** 향후 사용자/비용이 커지면 재검토.

## 7. 만약 3순위(네이티브 VAD)를 진행한다면
1. ADR 추가: "STT 전 VAD 전처리 단계" 결정.
2. 네이티브 모듈: Android(MediaExtractor+MediaCodec→PCM→RMS), iOS(AVAssetReader→RMS). 반환: `hasSpeechProbability` 또는 `peakDb`.
3. config plugin으로 모듈 등록(현 widget 플러그인 패턴 재사용).
4. `handleStt` 진입부에서 호출 → 임계 미만이면 `skipped` 후 종료.
5. UI: "음성 없음" 표시 + 상세 화면 "STT 재시도"로 강제 전송 경로 유지.
6. 검증: 무음/저음/정상/잡음 클립 4종으로 임계 튜닝, 오판율 측정.

## 결론
- **지금 단계에선 VAD 전송-차단까지는 과투자.** 환각의 핵심 피해(가짜 텍스트 저장)는 이미 사후 필터로 막았고, 비용 절감 효과는 개인용 규모에서 미미하다.
- 가장 비용효율적 다음 수: **"필터 후 빈 본문 → skipped + '음성 없음' UI"** 마감 처리(1순위). voice 모드 무음까지 원천 차단이 꼭 필요해질 때 네이티브 VAD(3순위)를 도입한다.
